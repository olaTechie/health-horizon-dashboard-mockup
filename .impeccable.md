# Impeccable — Project Design Context

This file captures the design context for the **Shell Health Horizon Dashboard** prototype so that future `/impeccable` invocations (`craft`, `typeset`, `layout`, `polish`, `harden`, etc.) operate from a shared, opinionated brief rather than re-deriving direction every time.

## Design Context

### Users

The dashboard is consumed by senior Shell Health leadership and adjacent governance audiences:

- **Shell Health executives** (PPA pillar leads, lead health expert authors of the quarterly Brief)
- **HSSE Policy leads** at Shell HQ
- **Regional health leads** in business segments (Upstream, Integrated Gas/LNG, Downstream/Chemicals, Renewables)
- **RFI evaluators** at Shell Supply Chain Commercial — the first-impression audience. They open the live URL once, on a desktop, between other vendor responses. They will not click everything, so density-on-first-paint matters more than long exploration paths.

The job-to-be-done is translating continuous global health signal volume — Mpox, AMR, NIOSH heat-stress rulemaking, climate–health interfaces, psychosocial workforce risk — into PPA-tagged, SEAM-mapped executive decisions within the 24–72h decision window for Action-tier alerts. Users do not want to be entertained. They want to know within seconds:

1. Is anything Action-tier today, and what does it ask me to decide?
2. Is the methodology defensible if I'm asked by HSSE board?
3. Where on Shell's footprint does this matter?

### Brand Personality

Three words: **Operational · Forensic · Live**, holding hands with **Premium · Strategic · Considered**.

The hybrid is intentional. Pure Palantir-ops without the consultancy gloss reads as a tool for analysts, not a deliverable to executives. Pure McKinsey-deck without the live operational register reads as last quarter's PowerPoint, not a continuously updated intelligence platform. The user's product needs both registers in tension:

- **Ops register dominates** in the dark theme, on `/map`, in the signal explorer table, in the alert ticker — this is where the "Living Intelligence Architecture" of the proposal must visibly live.
- **Editorial register dominates** in the light theme, on `/briefs`, in the signal detail's recommended-action call-out, in the dashboard hero — this is where the methodology rigor and executive framing must be readable to a non-technical evaluator.

### Aesthetic Direction

**Primary references** (the user explicitly selected these):
- **Palantir Foundry** — operational density, color-coded entity types, mono accents, asymmetric panel composition.
- **Bloomberg Terminal** — amber/red signal hierarchy, time-stamped everything, no decorative whitespace, function before flourish.

**Anti-reference** (the user explicitly rejected):
- **Corporate enterprise IT portal** (SAP/Oracle/Workday). Slab buttons, flat low-contrast greys, dated card grids, the visual register of "internal compliance tool nobody opens twice" — must never read this way.

**Theme posture:**
- **Editorial (light)** is the default for cognitive ease on first paint and for reading the long-form Brief. Warm off-white canvas, generous editorial type for headlines, restrained density.
- **Ops (dark)** is the situation-room mode for the geographic and explorer views. Near-black blue canvas, panel-on-panel composition, mono-coded data, restrained but present accent color. Should feel *more* invested than the light theme — this is where the "live intelligence" claim is cashed.

**Color encoding:**
- Tier (Watch / Attention / Action) is the **spine of the visual hierarchy**. Cool blue → amber → red. Action-tier signals are the only elements that earn motion (pulsing rings).
- PPA pillars (Promote / Protect / Access) appear as accent chips with their own colorways (sage / slate-violet / terracotta) — they tag and disambiguate but never act as primary fills. They must remain readable in both themes.
- No Shell brand colors (red/yellow/Pecten). RFI clause forbids it. Palette is deliberately non-Shell.

### Design Principles

1. **Tier hierarchy beats decoration.** Every visual decision must reinforce the Watch → Attention → Action spine. If a flourish weakens it, the flourish loses.
2. **Methodology is the design.** Triangulation count, source-type icons, evidence-trail timeline, confidence rationale — these earn pixels because they are why an evaluator will trust the platform. Decorative chrome that does not serve methodology is cut.
3. **Time-coding everywhere.** Every datum has a "last updated" stamp. Mono small-caps for timestamps. The product must feel *alive*, not retrospective. This is the single hardest thing to convey to an RFI reader and worth disproportionate craft investment.
4. **Density is a feature.** Executives read fast and laterally. Tight 4px grid, panel-on-panel composition, no template-grid card walls. Whitespace is rationed to where it carries hierarchy work.
5. **Both themes are first-class.** No "mode A is the real one and mode B is a swap." Editorial and Ops each get their own type display, surface tints, and motion register. Theme toggle is a posture switch, not a preference.

## Implementation State (as of v0.1.0)

- **Stack:** Next.js 15 (static export), TypeScript strict, Tailwind v4, MapLibre GL, Recharts, Framer Motion. Self-hosted variable fonts.
- **Type stack currently shipped:** Fraunces (editorial display) + Inter (UI + ops display via optical-size axis) + JetBrains Mono (data). All three are on the impeccable reflex-fonts-to-reject list. Re-evaluating this is a candidate for a future `/impeccable typeset` pass — but the current pairing is functioning and self-consistent, so any change must come with clear improvement, not novelty for its own sake.
- **Color tokens:** OKLCH-equivalent hex values defined in `app/globals.css` as CSS variables under `:root, [data-theme="editorial"]` and `[data-theme="ops"]`. Tokens to revisit if a future pass converts to OKLCH directly.
- **Constraints baked in:** illustrative-prototype footer chip on every page; no Pecten; basePath `/health-horizon-dashboard-mockup` for GitHub Pages.
- **Live URL:** https://olatechie.github.io/health-horizon-dashboard-mockup/

## Anti-Slop Checklist (Project-Specific)

When invoking any `/impeccable` design skill on this project, do not add:
- Side-stripe colored borders on cards/list items (BAN 1 — particularly tempting on signal cards; do not introduce).
- Gradient text on the "Global Health Horizon" headline or anywhere else (BAN 2).
- Glassmorphism on map overlays or filter panels.
- Sparklines as decoration (the dashboard's KPI sparklines are functional and labeled — keep them, but do not extend the pattern).
- Generic-template card grids with `icon + heading + text + arrow`.
- Centered hero metrics with gradient accents.
- The default AI palette (cyan-on-dark, purple-to-blue gradients, neon accents).
